## 1. Goal

Add a persistent, append-only `activity_events` table to record audit events across domains. Initial scope covers the user management and budget management domains. The table is designed to be extended to future domains without schema migrations.

---

## 2. Design Decisions

### 2.1 Event log, not state history

Each row records an action that occurred ("user was deactivated") rather than a state snapshot ("user had these attributes from T1 to T2"). The audit use case is satisfied by the event log shape.

### 2.2 Single table with generic entity reference (Approach C)

One `activity_events` table with a `domain` VARCHAR discriminator. Entity references are generic (`entity_type` + `entity_id` as strings) rather than domain-specific FK columns. Rationale:

- Adding a new domain requires zero schema migration â€” emit new `domain`/`entity_type`/`event_type` values.
- No nullable FK column proliferation as domains are added.
- Row width stays constant regardless of how many domains are active.
- Trade-off accepted: no DB-level referential integrity on `entity_id`. The codebase already uses this pattern (`ProjectSpendTracking` has no cascade FKs on its entity columns for the same reason â€” event history must survive entity deletion).

### 2.3 VARCHAR discriminators, no PG ENUMs

Project convention. PG ENUMs have caused operational pain in this codebase (see `alembic/env.py` `alembic_postgresql_enum` patch). All discriminator columns use VARCHAR with application-layer validation via Pydantic `Literal` types.

### 2.4 Transactional coupling

The repository `insert` method receives the caller's `Session`. Event rows are committed in the same transaction as the business mutation. A failed mutation rolls back both the business change and the event row, preventing phantom audit entries for operations that never completed.

### 2.5 Append-only

Rows in `activity_events` are never updated or deleted by application code. Consistent with `skill_events` and `project_spend_tracking`.

## 3. Table Schema

**File:** `src/codemie/service/activity/activity_models.py`

```python
class ActivityEvent(SQLModel, table=True):
    __tablename__ = "activity_events"

    id: str = Field(default_factory=lambda: str(uuid4()), primary_key=True)

    domain: str = Field(sa_column=Column(VARCHAR(64), nullable=False))
    event_type: str = Field(sa_column=Column(VARCHAR(128), nullable=False))

    entity_type: Optional[str] = Field(
        default=None, sa_column=Column(VARCHAR(64), nullable=True)
    )
    entity_id: Optional[str] = Field(
        default=None, sa_column=Column(VARCHAR(36), nullable=True)
    )

    actor_id: Optional[str] = Field(
        default=None,
        sa_column=Column(
            VARCHAR(36),
            ForeignKey("users.id", ondelete="SET NULL"),
            nullable=True,
        ),
    )

    attributes: Optional[dict] = Field(
        default=None, sa_column=Column(JSONB, nullable=True)
    )

    created_at: datetime = Field(
        sa_column=Column(
            TIMESTAMP(timezone=True),
            server_default=func.now(),
            nullable=False,
        )
    )

    __table_args__ = (
        Index(
            "ix_activity_events_entity_type_entity_id",
            "entity_type", "entity_id",
            postgresql_where=text("entity_id IS NOT NULL"),
        ),
        Index(
            "ix_activity_events_actor_id_created_at",
            "actor_id", "created_at",
            postgresql_where=text("actor_id IS NOT NULL"),
        ),
        Index(
            "ix_activity_events_domain_created_at",
            "domain", "created_at",
        ),
    )
```

### Column notes

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `id` | VARCHAR(36) | NOT NULL | UUID string PK, generated by application |
| `domain` | VARCHAR(64) | NOT NULL | Discriminator: `user_management`, `budget_management`, â€¦ |
| `event_type` | VARCHAR(128) | NOT NULL | Dot-notation: `user.created`, `budget.user_assignment.created`, â€¦ |
| `entity_type` | VARCHAR(64) | NULL | Entity kind: `user`, `budget`, â€¦ Null for domain-level events |
| `entity_id` | VARCHAR(36) | NULL | Entity PK as string. Null iff `entity_type` is null |
| `actor_id` | VARCHAR(36) | NULL | FK â†’ `users.id` ON DELETE SET NULL. Null for system-initiated events |
| `attributes` | JSONB | NULL | Domain-specific payload: before/after state, extra context |
| `created_at` | TIMESTAMPTZ | NOT NULL | Server-default `now()`, set by DB |

`entity_type` and `entity_id` are a **pair**: both set or both null. This invariant is enforced at the application layer, not the DB.

class ActivityEventCreate(BaseModel):
    domain: str
    event_type: str
    entity_type: Optional[str] = None
    entity_id: Optional[str] = None
    actor_id: Optional[str] = None
    attributes: Optional[dict] = None

    @model_validator(mode="after")
    def entity_pair_must_be_complete(self) -> "ActivityEventCreate":

        return self

## 6. Repository

**File:** `src/codemie/service/activity/activity_repository.py`

Abstract base + concrete implementation, following the `SkillEventRepository` pattern. Session is passed in by the caller to enable transactional coupling.

```python
class ActivityEventRepository(ABC):
    @abstractmethod
    def insert(self, event: ActivityEventCreate, session: Session) -> ActivityEvent: ...

    @abstractmethod
    def find_by_entity(
        self,
        entity_type: str,
        entity_id: str,
        *,
        limit: int,
        offset: int,
        session: Session,
    ) -> list[ActivityEvent]: ...

    @abstractmethod
    def find_by_actor(
        self,
        actor_id: str,
        *,
        limit: int,
        offset: int,
        session: Session,
    ) -> list[ActivityEvent]: ...

    @abstractmethod
    def find_by_domain(
        self,
        domain: str,
        *,
        from_dt: datetime,
        to_dt: datetime,
        limit: int,
        offset: int,
        session: Session,
    ) -> list[ActivityEvent]: ...


ActivityEventRepositoryImpl: ActivityEventRepository = SQLActivityEventRepository()
```

All list methods are bounded (`limit` required). No unbounded queries per database-optimization guide.
